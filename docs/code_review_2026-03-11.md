## 主要发现

### 1) `listTaskRuntimeStats` 在读锁内执行回调，可能放大锁持有窗口（低）
**现象**
- 当前 `listTaskRuntimeStats()` 持有 `taskRuntimeStatsMu.RLock` 时直接执行回调 `fn()`。
- 若未来回调逻辑变重或发生阻塞，会延长读锁持有时间，间接影响注册/注销（写锁）时延。

**建议**
- 先复制 `task->fn` 快照，释放读锁后再执行回调，降低锁竞争风险。

### 2) TCP 实际转发测试仍依赖“立即可连”假设，存在脆弱窗口（低）
**现象**
- `TestForwardMatrixTCPToTCP_Actual` 在 `UpdateCache` 后立即 `Dial` receiver 端口。
- 在调度抖动较大的环境中，该模式理论上存在 listener 尚未完全就绪的瞬时失败可能。

**建议**
- 在测试中增加短时重试或显式 ready 探针，以减少环境抖动导致的偶发失败。
