# 第一次全局代码审查

## 1. 审查背景

### 当前仓库状态

- 当前工作分支：`master`。
- 当前项目形态：Go 转发引擎，主链路为 `receiver -> selector -> task_set -> task -> pipeline -> sender`。
- 当前配置形态：启动入口使用双配置模式，`system config` 承载 `control`、`logging`、`business_defaults`，`business config` 承载可热重载业务拓扑。
- 最近已合并的修复重点集中在 `config/runtime` validation、runtime 边界和部分 sender/receiver/task 行为；本轮不以局部补丁为目标，而是做全局收口。
- 当前文档状态：`README.md` 已有快速入口，`docs/` 下存在配置、运行时、架构、运维、协议、pipeline、observability 等多份文档，但存在主题重复、历史说明残留、部分章节编号/职责边界不清的问题。

### 本轮审查目标

- 对整个项目自有代码、测试、benchmark、README、`docs/` 和 example config 做一轮系统审查。
- 修复确认为真实问题的代码缺陷，避免只做风格性调整。
- 梳理全局代码注释，补齐核心结构、并发/生命周期、reload/stop/cleanup、热路径边界和配置语义映射处的必要说明。
- 整合 README 和 `docs/`，删除过时、重复或与代码不一致的内容，让新读者通过当前文档理解当前实现。
- 将分析、计划、执行过程持续沉淀在本文档中，后续所有执行以本文档为准。

### 约束条件

- 不影响性能：热路径不得引入新的 map/slice 分配、粗锁、额外复制、`fmt.Sprintf`、JSON 序列化、重型抽象或无必要 `defer`。
- 不做大规模重构：优先小步修复，保持当前架构方向和模块边界。
- 先验证再修复：代码问题必须能解释为真实风险；文档问题必须能说明与代码、示例或入口职责的偏差。
- 冷路径优先：确需增加开销时，应限制在配置加载、校验、构建、reload、错误处理或 shutdown 路径。

## 2. 审查范围

- `src/bootstrap`：参数解析、双配置加载、控制面拉取、文件监听、信号 reload、pprof/GC 日志、停机顺序。
- `src/config`：结构体注释、严格 JSON 解析、system/business merge、默认值、校验、example config 覆盖。
- `src/runtime`：Store 生命周期、全量替换、增量热更新、selector rebuild、stage cache、sender refs、receiver start/stop、dispatch fastpath。
- `src/task`：执行模型、in-flight 统计、StopGraceful、channel/pool/fastpath、pipeline 执行、route sender、traffic counter。
- `src/pipeline`：stage 编译语义、match/replace/route/file rewrite、热路径分配与 packet 生命周期。
- `src/selector`：compiled selector 快照、精确匹配、default task_set、nil selector 行为。
- `src/receiver`：UDP/TCP gnet、Kafka、SFTP、OSS、SkyDDS、local_timer、match key builder、Stop 幂等和异常路径。
- `src/sender`：UDP/TCP/Kafka/SFTP/OSS/SkyDDS、notifier、file path 安全、chunk 状态机、Close 幂等和错误语义。
- `src/logx`：logger 初始化、traffic counter、runtime stats 注册/注销、聚合线程生命周期、采样热路径。
- `src/skydds`：无 SDK stub、cgo wrapper、C/C++ bridge、batch/octet 语义、错误与资源释放边界。
- `README.md`：项目入口、最小示例、配置/文档索引、验证命令、SkyDDS 说明。
- `docs/`：全部 Markdown 文档和 SkyDDS PDF 的引用关系；重点整理 Markdown。
- `tests`：各模块 `*_test.go`，覆盖配置、runtime、receiver/sender、task、pipeline、selector、logx、SkyDDS。
- `benchmarks`：各模块 benchmark，包括 `src/runtime`、`src/task`、`src/pipeline`、`src/selector`、`src/logx`、`src/sender`、`src/receiver` 中现有 benchmark。
- example config：`configs/*.json`、`deploy/k8s/configmap.yaml` 中配置语义和文档一致性。

### P0：必须修

#### P0-1：README/docs 存在重复和局部不一致，影响新读者建立当前系统模型

- 问题描述：`docs/runtime-and-lifecycle.md`、`docs/runtime-sequence-and-flow.md`、`docs/technical-architecture.md`、`docs/architecture.md`、`docs/task-and-dispatch.md`、`docs/operations-manual.md` 多处重复解释启动、热重载、match key、execution model、observability；`docs/receivers-and-senders.md` 出现重复/错序标题（两个 SkyDDS receiver 小节，OSS 小节编号夹在其中）；`docs/roadmap.md` 保留“待补项/局限/演进”类历史内容，容易与当前收口目标冲突。
- 风险：读者可能不知道哪份文档是权威入口，配置和运行时语义被重复描述后容易漂移；重复章节会导致后续维护继续产生不一致。
- 涉及模块：`README.md`、`docs/architecture.md`、`docs/technical-architecture.md`、`docs/runtime-and-lifecycle.md`、`docs/runtime-sequence-and-flow.md`、`docs/receivers-and-senders.md`、`docs/operations-manual.md`、`docs/roadmap.md`、`docs/operations.md`。
- 为什么是实际问题：文档已经是项目的运行入口和配置入口；重复/错序/历史规划会直接影响部署、热重载和排障操作，不是排版风格问题。
- 是否可能影响性能：文档整理本身不影响运行时性能。
- 预期修复方式：明确 README 为快速入口，`docs/configuration.md` 为配置权威入口，`docs/runtime-and-lifecycle.md` 为运行时权威入口，`docs/receivers-and-senders.md` 为协议权威入口，`docs/operations-manual.md` 为操作手册；合并或删减重复内容，移除 `roadmap` 中不适合作为当前实现说明的历史待办。

#### P0-2：必须先建立并持续维护本文档作为计划与执行记录

- 风险：执行过程无法审计，新增问题没有统一优先级，最终结果无法证明覆盖了全项目。
- 涉及模块：`docs/第一次全局代码审查.md`。
- 为什么是实际问题：这是本轮工作流约束，会影响交付完整性。
- 是否可能影响性能：无运行时影响。
- 预期修复方式：先创建本文档；每轮执行结束后追加“执行记录”，新增问题先写入问题清单再处理。

### P1：建议本轮修

#### P1-1：runtime 增量更新的部分失败路径需要继续核对回滚边界

- 问题描述：`applyBusinessDelta` 会按 selector/task_set、sender、pipeline、task、receiver 顺序逐步更新 Store。部分步骤已在构建阶段提前失败或关闭已构建对象，但仍需要重点核对 task 重建失败、receiver selector rebuild 失败、sender 替换后 task add 失败等路径是否会留下半更新状态。
- 风险：热重载失败后旧配置可能不再完整生效，或留下新 sender/receiver/stage cache/counter 的残留引用。
- 涉及模块：`src/runtime/update_cache.go`、`src/runtime/store.go`、`src/task/task.go`、`src/logx/traffic_agg.go`。
- 为什么是实际问题：热重载失败后的“旧配置继续服务”是文档和运维语义承诺；半更新不是风格问题。
- 是否可能影响性能：修复应限定在 reload 冷路径，不应改变 dispatch/task 热路径。
- 预期修复方式：补充针对失败路径的单元测试；如发现真实残留，采用冷路径预构建、延迟提交或显式回滚修复。

#### P1-2：route sender miss 当前只有 warn 日志，缺少可观测计数或更清晰文档边界

- 问题描述：`task.processAndSend` 在 `Meta.RouteSender` 不属于当前 task senders 时只打 warn 并丢弃；代码中有 TODO 提示增加 `route_sender_miss` 计数器。
- 风险：生产中 route stage 与 task.senders 漂移时，流量被丢弃但只靠日志检索发现；高流量下 warn 可能被采样/滚动淹没。
- 涉及模块：`src/task/task.go`、`src/logx/traffic_agg.go`、`docs/configuration.md`、`docs/task-and-dispatch.md`、`docs/pipeline.md`。
- 为什么是实际问题：这是实际丢包路径的观测缺口，不是命名或格式问题。
- 是否可能影响性能：如果增加计数，不能在热路径引入 map 分配或粗锁；优先考虑复用既有 traffic counter 或先文档化为 P2。
- 预期修复方式：先核对现有统计模型；若能零额外分配地记录 miss，则补实现和测试；否则本轮至少将语义写入权威文档并保留为 P2。

#### P1-3：核心对外类型和热路径函数注释仍不均衡

- 问题描述：`src/selector`、`src/pipeline`、`src/runtime/types.go`、部分 receiver/sender 构造与 Close/Stop、SkyDDS cgo wrapper 的注释密度不均；部分注释解释“做什么”，但没有解释并发、热路径或状态转换约束。
- 风险：后续改动容易在 selector 快照、packet release、receiver stop、file transfer 状态机、SkyDDS batch 语义上误改。
- 涉及模块：`src/runtime`、`src/task`、`src/pipeline`、`src/selector`、`src/receiver`、`src/sender`、`src/logx`、`src/skydds`。
- 为什么是实际问题：这些位置是维护边界和生命周期边界，注释缺失会增加误改概率；不是为了注释覆盖率。
- 是否可能影响性能：注释无运行时影响。
- 预期修复方式：只在核心结构、关键字段、并发/生命周期、热路径函数和不易理解的边界补充短注释；删除或修正过时注释。

#### P1-4：README 和 docs 的职责边界需要收口

- 问题描述：README 目前既包含最小示例、配置关键规则、文档索引和 SkyDDS 构建；docs 中又重复展开相同内容，且 `operations.md` 只是薄入口，`operations-manual.md` 是长文档。
- 风险：入口层级不清，读者可能跳过权威文档或阅读重复内容。
- 涉及模块：`README.md`、`docs/operations.md`、`docs/operations-manual.md`、`docs/configuration.md`、`docs/skydds.md`。
- 为什么是实际问题：文档导航直接影响部署和配置正确性。
- 是否可能影响性能：无运行时影响。
- 预期修复方式：README 保留快速启动、最小示例、文档地图和验证命令；docs 承载细节；`operations.md` 保留为运维入口并指向操作手册；删除 README 中过深的重复说明。

#### P1-5：receiver/sender 文件链路异常路径需要再核对状态机与文档一致性

- 问题描述：SFTP/OSS sender 涉及 chunk 校验、重复/乱序、commit、notify_on_success、失败 abort；docs 中声明“commit 成功后才通知”“通知失败会返回错误”等，需要与实际代码一致。
- 风险：文件已提交但通知失败、abort 失败、重复 chunk 或乱序缓存上限处理不清，会影响幂等和运维补偿。
- 涉及模块：`src/sender/sftp.go`、`src/sender/oss.go`、`src/sender/notifier.go`、`src/receiver/sftp.go`、`src/receiver/oss.go`、`docs/receivers-and-senders.md`、`docs/configuration.md`。
- 为什么是实际问题：文件链路是有状态协议，错误语义直接影响数据一致性。
- 是否可能影响性能：修复应限于错误/commit/close 冷路径；不能增加每 chunk 热路径重型开销。
- 预期修复方式：审查并补充测试；如行为与文档不一致，以代码正确性优先修复或以当前代码为准修正文档。

#### P1-6：bootstrap 配置加载/重载文档与代码边界需要统一

- 问题描述：`loadConfigPair` 在控制面 API 存在时用远端 business 覆盖本地 business；热重载仍检查 system 基线稳定。文档多处描述该流程，但分散且重复。
- 风险：使用者可能误以为本地 business 文件在配置中心启用后仍作为业务来源，或误改 system 期待热重载生效。
- 涉及模块：`src/bootstrap/run.go`、`src/app/runtime.go`、`docs/configuration.md`、`docs/runtime-and-lifecycle.md`、`docs/operations-manual.md`。
- 为什么是实际问题：配置来源和热重载边界是运行正确性的入口。
- 是否可能影响性能：文档/注释为主，无热路径影响。
- 预期修复方式：保留一处权威流程说明，其他文档只引用结论；必要时补启动入口注释。

#### P1-7：UDP sender 构造失败时已打开分片 socket 需要清理

- 问题描述：`NewUDPUnicastSender` / `NewUDPMulticastSender` 会循环打开多个分片 socket；如果中途某个分片创建失败，前面已成功打开的 socket 没有统一关闭。
- 风险：构造失败后泄漏 fd/端口绑定，热重载或启动失败重试时可能放大为端口占用或 fd 泄漏。
- 涉及模块：`src/sender/udp_unicast.go`、`src/sender/udp_multicast.go`。
- 为什么是实际问题：这是错误路径资源泄漏，不是风格问题。
- 是否可能影响性能：只影响构造失败冷路径，不影响 `Send` 热路径。
- 预期修复方式：构造循环中 `ensureConn` 返回错误时调用 `Close(context.Background())` 清理已打开分片。

#### P1-8：SkyDDS cgo reader/writer Close 与读写调用并发时缺少 C 指针保护

- 问题描述：`cgoReader` / `cgoWriter` 的 `Close` 会关闭并置空 C 指针；`Wait` / `Drain` / `Write` / `WriteBatch` 同时使用该指针，真实 SDK 构建下存在 Close 与读写并发窗口。
- 风险：Stop/Close 与 Start/Send 并发时可能出现 use-after-close 或空指针竞态。
- 涉及模块：`src/skydds/skydds_cgo.go`、`src/receiver/skydds.go`、`src/sender/skydds.go`。
- 为什么是实际问题：这是跨 cgo 资源生命周期竞态，不是注释或风格问题。
- 是否可能影响性能：只影响 `-tags skydds` 构建；锁粒度限定为单次 C 调用和指针置空，不影响非 SkyDDS 主链路。
- 预期修复方式：为 cgo reader/writer 增加 mutex，所有 C 指针检查、C 调用、Close 置空都在同一锁内完成。

### P2：记录但可后续处理

#### P2-1：task pipeline 多包扩展路径当前有保守分配

- 问题描述：`task.processAndSend` 每条 pipeline 都创建 `next` slice；当前 stage 多数是过滤/改写，不常扩展多包。
- 风险：如果后续新增 fan-out stage，高吞吐下可能有额外分配；当前已有 benchmark 应作为守护。
- 涉及模块：`src/task/task.go`、`src/pipeline`、`src/runtime` benchmark。
- 为什么是实际问题：这是热路径性能关注点，但当前不一定构成回归或缺陷。
- 是否可能影响性能：任何修复都可能影响热路径，需要 benchmark 证明。
- 预期修复方式：本轮只记录和 benchmark；除非发现明显回归，不做结构性优化。

#### P2-2：`docs/roadmap.md` 是否保留需要产品/维护策略确认

- 问题描述：roadmap 文档包含“待补项”“可演进方向”等规划内容，和本轮“当前系统说明”目标不同。
- 风险：如果作为实现文档阅读，会误导当前能力；如果作为规划文档保留，则需要明确非权威。
- 涉及模块：`docs/roadmap.md`、README 文档索引。
- 为什么是实际问题：文档职责边界问题。
- 是否可能影响性能：无。
- 预期修复方式：本轮优先降级为“后续演进记录”或移出主要索引；如内容完全过时则删除。

#### P2-3：SkyDDS cgo/C++ bridge 的深层错误映射依赖 SDK 环境，当前只能做有限验证

- 问题描述：无 SkyDDS SDK 时只能跑 stub 和 Go 层测试；cgo/C++ bridge 的真实 SDK 行为无法在普通环境完全验证。
- 风险：SDK 错误码、buffer 上限、close 幂等在真实环境仍可能存在差异。
- 涉及模块：`src/skydds`、`src/receiver/skydds.go`、`src/sender/skydds.go`、`docs/skydds.md`。
- 为什么是实际问题：外部 SDK 绑定属于运行边界，无法完全由 Go 单测覆盖。
- 是否可能影响性能：本轮不改 cgo 热路径，验证限制需记录。
- 预期修复方式：跑无 SDK stub 测试；如无法跑 `-tags skydds`，在执行记录中明确环境限制。

## 5. 性能守护原则

### 热路径

- `runtime.dispatchToSelector` / `dispatch`：receiver 入站后的 selector lookup 和 task fan-out。
- `task.Handle` / `task.runPacket` / `task.processAndSend` / `task.sendToSender`：执行模型入队、pipeline、sender fan-out、traffic counter。
- `pipeline.Process` / `mapStage` / `MatchOffsetBytes` / `ReplaceOffsetBytes` / `RouteSenderByOffsetBytes`：payload 匹配、改写和路由。
- receiver stream：
  - UDP/TCP gnet `OnTraffic`/frame loop。
  - Kafka poll record loop。
  - SFTP/OSS chunk 下发。
  - SkyDDS wait/drain/poll batch。
- sender send：
  - UDP/TCP/Kafka `Send`。
  - SFTP/OSS chunk accept/upload state machine。
  - SkyDDS octet/batch send。
- `logx.TrafficCounter.AddBytes` 和 runtime stats 聚合采样。

### 禁止引入的热路径开销

- 新的 per-packet map 分配或 slice 分配。
- `fmt.Sprintf`、JSON 序列化、`strings.Builder` 等格式化构造。
- 粗粒度锁覆盖 packet 处理、selector match、sender send。
- 额外 payload 复制或重复 packet clone。
- 无必要 `defer`。
- 重型接口层或反射。

### 允许增加开销的位置

- 配置加载、`ApplyDefaults`、`Validate`。
- pipeline/selector/receiver/sender/task 构建阶段。
- reload 计划生成、预构建、提交、回滚。
- Stop/Close/cleanup/abort。
- 错误路径和日志路径。

### 若必须增加少量开销

- 必须证明开销在冷路径、错误路径或 reload 路径。
- 如果触及热路径，必须补 benchmark/benchmem，说明 alloc 和 ns/op 变化。
- 优先复用现有结构和预编译快照，不在 per-packet 路径做通用解析。
